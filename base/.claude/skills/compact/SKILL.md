# /compact (Strategic Compact Guide)

**사용법**: `/compact` (이 스킬은 컴팩션 시점 판단을 도와줍니다)

## 왜 전략적 컴팩션인가?

자동 컴팩션은 임의 시점에 트리거됩니다:
- 종종 작업 중간에 발생 → 중요한 컨텍스트 유실
- 논리적 작업 경계를 인식하지 못함
- 복잡한 멀티스텝 작업 중단 가능

**전략적 컴팩션**은 논리적 경계에서:
- 탐색 → 구현 전환 시: 리서치 컨텍스트 정리, 구현 계획만 유지
- 마일스톤 완료 후: 다음 단계를 위한 프레시 스타트
- 주요 컨텍스트 전환 전: 이전 작업 컨텍스트 정리

## 컴팩션 판단 가이드

| 단계 전환 | 컴팩트? | 이유 |
|----------|---------|------|
| 리서치 → 계획 수립 | ✅ Yes | 리서치 컨텍스트는 양이 많음; 계획이 정제된 결과물 |
| 계획 → 구현 | ✅ Yes | 계획은 todo.md에 저장됨; 코드 작성 공간 확보 |
| 구현 → 테스트 | ⚠️ Maybe | 테스트가 최근 코드 참조하면 유지; 포커스 전환이면 컴팩트 |
| 디버깅 → 다음 기능 | ✅ Yes | 디버그 트레이스가 관련 없는 작업 오염 |
| 구현 중간 | ❌ No | 변수명, 파일 경로, 부분 상태 유실 비용이 큼 |
| 실패한 접근 후 | ✅ Yes | 막다른 추론 정리 후 새 접근 시도 |

## 컴팩션 후 유지/유실되는 것

| 유지됨 | 유실됨 |
|--------|--------|
| CLAUDE.md 지시사항 | 중간 추론 및 분석 |
| tasks/todo.md 내용 | 이전에 읽은 파일 내용 |
| MEMORY.md 파일 | 멀티스텝 대화 컨텍스트 |
| Git 상태 (커밋, 브랜치) | 도구 호출 이력 |
| 디스크 파일 | 구두로 전달된 사용자 선호 |

## 모범 사례

1. **계획 완료 후 컴팩트** — todo.md에 계획 확정 후 프레시 스타트
2. **디버깅 후 컴팩트** — 에러 해결 컨텍스트 정리
3. **구현 중에는 컴팩트 금지** — 관련 변경사항 컨텍스트 유지
4. **컴팩트 전 저장** — 중요 컨텍스트를 파일이나 memory에 저장
5. **요약과 함께 컴팩트** — `/compact 다음 작업: auth 미들웨어 구현`

## 규칙

- 이 스킬은 판단을 돕는 가이드이며, 실제 컴팩션은 사용자가 `/compact` 명령으로 실행
- 강제 컴팩션하지 않음 — 제안만
